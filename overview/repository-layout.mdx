The ABP Framework is developed as a single Git monorepo whose root contains sixteen top-level folders and seven build/config files. This page is the field guide to that tree: what lives where, how `framework/` relates to `modules/`, where the `.slnx` solutions and `.abpsln` ABP-Studio solutions sit, and how the ~169 `Volo.Abp.*` packages under `framework/src/` are clustered by prefix. Treat it as the index you reach for before opening a file.

The repository is rooted at the GitHub slug `abpframework/abp` (declared in `Directory.Build.props` via `<RepositoryUrl>https://github.com/abpframework/abp/</RepositoryUrl>`). Every top-level folder is independent enough to open in its own IDE window â€” there is no single uber-solution that pulls in every project. Instead, `build/common.ps1` lists the solutions that the CI script walks in order. See [/overview/solution-and-build](/overview/solution-and-build) for the build orchestration and [/overview/architecture](/overview/architecture) for the nine logical layers those solutions implement.

<Note>
  The numbers in this page (e.g. "169 packages under `framework/src/`",
  "38 `Volo.Abp.AspNetCore.*` packages") were extracted by listing the
  directories directly â€” they will drift slightly between releases as new
  providers (Bunny CDN storage, TickerQ, etc.) land. They are accurate as of
  the `dev` branch snapshot used to author this wiki.
</Note>

## `framework/` â€” the core framework

`framework/` is the heart of the repo. It contains:

- `framework/Volo.Abp.slnx` â€” modern XML solution file listing every `src/` and `test/` project.
- `framework/Volo.Abp.abpsln` â€” ABP Studio solution descriptor (JSON, `template: empty`, references `Volo.Abp.abpmdl`).
- `framework/Volo.Abp.abpmdl` â€” ABP Studio module descriptor.
- `framework/Volo.Abp.sln.DotSettings` â€” ReSharper team-shared settings.
- `framework/NuGet.md` â€” the README that ends up packed into every `Volo.Abp.*` `.nupkg`.
- `framework/src/` â€” 169 NuGet-shipped packages, see the next section.
- `framework/test/` â€” 95+ xUnit test projects mirroring `src/` (e.g. `Volo.Abp.Auditing.Tests/`, `Volo.Abp.EntityFrameworkCore.Tests/`, plus the cross-cutting `AbpTestBase/`).

The framework has zero dependency on the `modules/` folder â€” application modules are downstream consumers.

### Inside a typical `Volo.Abp.*` package folder

Every package directory under `framework/src/` follows the same shape. Using `framework/src/Volo.Abp.EventBus/` as an example:

| File / folder | Purpose |
|---|---|
| `Volo.Abp.EventBus.csproj` | SDK-style csproj that imports `../../common.props` and pulls central package versions. |
| `Volo.Abp.EventBus.abppkg` | ABP Studio package descriptor (referenced from `.abpsln` / `.abpmdl`). |
| `Volo.Abp.EventBus.abppkg.analyze.json` | Per-package analyzer config (packed as content). |
| `AbpEventBusModule.cs` | The single `AbpModule` subclass; the entry point for `[DependsOn]`. |
| `Volo/Abp/EventBus/...` | Source tree using ABP's `Volo.Abp.X` namespace convention. |
| `Localization/Resources/AbpEventBus/...` | JSON localization (if the package emits user-facing strings). |

The framework convention is "one module per package, one package per assembly" â€” see [/core/modularity-and-modules](/core/modularity-and-modules).

## Configuration files at the repo root

The repo root contains six MSBuild / SDK control files that propagate to every project.

| File | Effect |
|---|---|
| `Directory.Build.props` | Auto-imported by MSBuild. Detects test projects by name (`*.Tests`, `*.TestBase` under a `test` path) and conditionally adds `coverlet.collector`. |
| `Directory.Packages.props` | Enables Central Package Management (`<ManagePackageVersionsCentrally>true</ManagePackageVersionsCentrally>`) and lists every `PackageVersion`. See [/overview/tech-stack-and-dependencies](/overview/tech-stack-and-dependencies). |
| `common.props` | Imported manually by every `Volo.Abp.*` `.csproj`. Sets `<LangVersion>latest</LangVersion>`, `<Version>10.2.0-rc.3</Version>`, `<LeptonXVersion>5.2.0-rc.3</LeptonXVersion>`, license `LGPL-3.0-only`, NuGet metadata, source-link, and `*.abppkg(.analyze.json)` content packing. |
| `common.test.props` | Imported by test projects. Same lang version + suppresses assembly metadata attributes. |
| `configureawait.props` | Imports the `ConfigureAwait.Fody` weaver in Release builds â€” strips `.ConfigureAwait(false)` boilerplate from async methods. |
| `global.json` | Pins `sdk.version` to `10.0.100` with `rollForward: latestFeature`. |
| `NuGet.Config` | Single feed: `nuget.org`. Nightly builds rely on a separate MyGet feed configured per-CI. |
| `latest-versions.json` | Release manifest read by abp.io to render version selectors. |
| `codecov.yml` | Codecov integration thresholds. |
| `common.DotSettings` | Team-shared ReSharper code style. |
| `delete-bin-obj.ps1` | Recursively deletes `bin/` and `obj/` â€” useful before a clean rebuild. |

<Warning>
  `Directory.Build.props` and `Directory.Packages.props` are picked up
  automatically by every nested project. Editing them touches the entire
  monorepo â€” use with care.
</Warning>
