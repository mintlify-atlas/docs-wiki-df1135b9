The ABP Framework is, at heart, an opinionated composition of well-known .NET libraries — and a parallel composition of npm packages on the Angular / MVC side. This page is the inventory: every `PackageVersion` declared in `Directory.Packages.props` matched to the `framework/src/Volo.Abp.*` packages that pull it in, plus the npm packages that drive `npm/ng-packs/` and `npm/packs/`. Use it to answer "what version of Autofac is ABP on?", "which package gives me MongoDB integration?", and "what versions of Angular ship with v10?".

For the props file mechanics that make central package management work, see [/overview/solution-and-build](/overview/solution-and-build); for the runtime that consumes these libraries, see [/overview/modularity-model](/overview/modularity-model). For .NET versions, `global.json` pins SDK `10.0.100` with `rollForward: latestFeature` — every package below targets `net10.0` unless otherwise noted.

## Where the numbers come from

Every version in this page is read directly from `Directory.Packages.props` at the repo root. That file enables Central Package Management with:

```xml
<PropertyGroup>
  <ManagePackageVersionsCentrally>true</ManagePackageVersionsCentrally>
  <CentralPackageFloatingVersionsEnabled>true</CentralPackageFloatingVersionsEnabled>
</PropertyGroup>
```

After that, ~180 `<PackageVersion Include="..." Version="..." />` entries fix the version of every transitive `PackageReference` in the monorepo. Individual `.csproj` files never carry a `Version=""` attribute.

<Note>
  The "Used by" column lists the ABP packages whose csproj declares a
  `<PackageReference>` for the row's library. If you need an exhaustive
  consumer list, `grep -rn "PackageReference.*Include=\"AutoMapper\"" framework/ modules/`
  inside the repo reveals every csproj.
</Note>

### `npm/packs/` — MVC bundles

`npm/packs/` ships static-asset NPM packages used by the MVC UI: `@abp/aspnetcore.mvc.ui`, `@abp/aspnetcore.mvc.ui.theme.basic`, `@abp/jquery`, `@abp/bootstrap`, etc. Each package wraps a vendored copy of a frontend library so the ASP.NET Core bundling pipeline (`Volo.Abp.AspNetCore.Mvc.UI.Bundling`) can resolve it relative to `wwwroot/libs/`.

The full list is rendered by Lerna at publish time — `npm/publish-mvc.ps1` runs `lerna publish` from `npm/packs/`.

## Version cross-walk

Knowing one ABP version, you can derive the others:

| What | Where | Example for v10.2.0-rc.3 |
|---|---|---|
| Framework / NuGet packages | `common.props` → `<Version>` | `10.2.0-rc.3` |
| LeptonX theme | `common.props` → `<LeptonXVersion>` | `5.2.0-rc.3` |
| Angular libs | `npm/ng-packs/package.json` → `version` of `@abp/*` | `~10.2.0-rc.3` |
| MVC NPM packs | `npm/packs/*/package.json` | `~10.2.0-rc.3` |
| .NET SDK | `global.json` → `sdk.version` | `10.0.100` |
| Microsoft.* runtime | `Directory.Packages.props` (most entries) | `10.0.2` |
| Public version manifest | `latest-versions.json` | top entry |

  Run `pwsh ./npm/preview-publish.ps1` against a local Verdaccio to verify
  the npm package versions match the NuGet `<Version>` before releasing —
  out-of-sync NPM and NuGet versions confuse the CLI's "update solution"
  workflow.
</Tip>

## Pinning vs floating

ABP pins exact versions almost everywhere (e.g. `Version="10.0.2"`, not `[10.0.*]`). The reasons:

1. **Reproducible CI builds** — `nupkg/0/` contains the same bits on every agent.
2. **Predictable consumer behaviour** — `dotnet add package Volo.Abp.Identity` always resolves the same transitive set.
3. **Single point of upgrade** — bumping EF Core means one PR touching one line in `Directory.Packages.props`.

The single concession is `CentralPackageFloatingVersionsEnabled=true` in `Directory.Packages.props`, kept on for future flexibility but not yet used by any `<PackageVersion>` entry.

## Looking up "which ABP package needs X?"

Three quick recipes:

1. **By library name** — search `framework/src/**/Volo.Abp.*.csproj` for `<PackageReference Include="LIBRARY_NAME"`. Results are short because central package management forbids per-csproj versions.
2. **By ABP package** — open the csproj. It will list its `<PackageReference>` items but no versions; cross-reference each name with `Directory.Packages.props` for the resolved version.
3. **By transitive graph** — `dotnet nuget why Volo.Abp.Identity Autofac` answers "why does my project end up referencing Autofac through Volo.Abp.Identity?".
