The Blob Storing Database module under `modules/blob-storing-database/src/` is one of the storage providers that plug into ABP Framework's [Blob Storing](/infrastructure/blob-storing) infrastructure. The blob-storing core defines `IBlobProvider`, `IBlobContainerFactory<TContainer>` and the `BlobContainerConfiguration` registration; this module ships `DatabaseBlobProvider`, a provider that writes blob bytes into a relational database table (`AbpBlobs`) or a Mongo collection (`AbpBlobs`).

This page walks through the source under `modules/blob-storing-database/src/` — the two aggregate roots `DatabaseBlobContainer` and `DatabaseBlob` declared in `Volo.Abp.BlobStoring.Database.Domain`, the `DatabaseBlobProvider` that bridges to the blob storing API, the `UseDatabase()` extension that wires a container to this provider, and the parallel EF Core / MongoDB persistence projects.

## Project layout

The module follows the same shape as [Audit Logging](/modules/audit-logging) — a pure infrastructure module with no application service:

```
modules/blob-storing-database/src/
├── Volo.Abp.BlobStoring.Database.Domain.Shared/    Consts, error codes, db properties
├── Volo.Abp.BlobStoring.Database.Domain/           DatabaseBlob, DatabaseBlobContainer, DatabaseBlobProvider
├── Volo.Abp.BlobStoring.Database.EntityFrameworkCore/ BlobStoringDbContext + EF repositories
├── Volo.Abp.BlobStoring.Database.MongoDB/          BlobStoringMongoDbContext + Mongo repositories
└── Volo.Abp.BlobStoring.Database.Installer/        NuGet meta-package
```

The module is a *provider* — the blob-storing core can address many providers (file system, Azure Blob, S3, Aliyun, MinIO, …); database storage is just one of them, and a host can mix providers per container.

## Aggregate roots

The module declares two aggregate roots in `Volo.Abp.BlobStoring.Database.Domain/Volo/Abp/BlobStoring/Database/`. Both are `IMultiTenant` so the `[TenantId]` data filter scopes them to the current tenant — a tenant cannot see another tenant's containers or blobs even on shared infrastructure.

### DatabaseBlobContainer

`DatabaseBlobContainer.cs` is a *named bucket* identified by a logical name (matching the `BlobContainerConfiguration` registration name in the host). It carries no payload of its own — it is purely the parent under which `DatabaseBlob` rows live:

```csharp
// modules/blob-storing-database/src/Volo.Abp.BlobStoring.Database.Domain/Volo/Abp/BlobStoring/Database/DatabaseBlobContainer.cs
public class DatabaseBlobContainer : AggregateRoot<Guid>, IMultiTenant
{
    public virtual Guid? TenantId { get; protected set; }
    public virtual string Name { get; protected set; }

    public DatabaseBlobContainer(Guid id, [NotNull] string name, Guid? tenantId = null)
        : base(id)
    {
        Name = Check.NotNullOrWhiteSpace(name, nameof(name), DatabaseContainerConsts.MaxNameLength);
        TenantId = tenantId;
    }
}
```

The `Name` corresponds to the container key used by the blob-storing API — for example calling `IBlobContainer.SaveAsync("file.png", stream)` on the default container resolves the container name to the `BlobContainerConfiguration` registered under `BlobContainerConfigurationProvider`'s default key.

### DatabaseBlob

`DatabaseBlob.cs` is the *blob payload*. Notice the `[DisableAuditing]` attribute on the `Content` property — auditing must not capture the raw bytes when entity history is enabled, because that would double-write every blob into the audit log:

```csharp
// modules/blob-storing-database/src/Volo.Abp.BlobStoring.Database.Domain/Volo/Abp/BlobStoring/Database/DatabaseBlob.cs
public class DatabaseBlob : AggregateRoot<Guid>, IMultiTenant
{
    public virtual Guid ContainerId { get; protected set; }
    public virtual Guid? TenantId { get; protected set; }
    public virtual string Name { get; protected set; }

    [DisableAuditing]
    public virtual byte[] Content { get; protected set; }

    public DatabaseBlob(Guid id, Guid containerId, [NotNull] string name, [NotNull] byte[] content, Guid? tenantId = null)
        : base(id)
    {
        Name = Check.NotNullOrWhiteSpace(name, nameof(name), DatabaseBlobConsts.MaxNameLength);
        ContainerId = containerId;
        Content = CheckContentLength(content);
        TenantId = tenantId;
    }

    public virtual void SetContent(byte[] content)
    {
        Content = CheckContentLength(content);
    }

    protected virtual byte[] CheckContentLength(byte[] content)
    {
        Check.NotNull(content, nameof(content));

        if (content.Length >= DatabaseBlobConsts.MaxContentLength)
        {
            throw new AbpException($"Blob content size cannot be more than {DatabaseBlobConsts.MaxContentLength} Bytes.");
        }

        return content;
    }
}
```

`DatabaseBlobConsts.MaxContentLength` defaults to `int.MaxValue` (≈2 GB) — the practical upper bound is determined by your database. SQL Server's `varbinary(max)` allows the full 2 GB; PostgreSQL `bytea` is similar; MongoDB documents are capped at 16 MB unless you use GridFS:

```csharp
// modules/blob-storing-database/src/Volo.Abp.BlobStoring.Database.Domain.Shared/Volo/Abp/BlobStoring/Database/DatabaseBlobConsts.cs
public static class DatabaseBlobConsts
{
    /// <summary>Default value: 256.</summary>
    public static int MaxNameLength { get; set; } = 256;

    /// <summary>Default value: <see cref="int.MaxValue"/> (2GB).</summary>
    public static int MaxContentLength { get; set; } = int.MaxValue;
}
```

<Warning>
The constants are *mutable static* by design — a host can lower `DatabaseBlobConsts.MaxContentLength` during startup to reject oversized uploads at the aggregate level, before they hit the database column constraint. The validation in `CheckContentLength` is part of the aggregate invariant, not a DB-only check.
</Warning>

## DatabaseBlobProvider — the BlobProviderBase implementation

`DatabaseBlobProvider` (`Volo.Abp.BlobStoring.Database.Domain/Volo/Abp/BlobStoring/Database/DatabaseBlobProvider.cs`) extends `BlobProviderBase` and implements the four mandatory operations: save, delete, exists, and get-stream. It is `ITransientDependency` so each operation runs in the caller's DI scope:

```csharp
// modules/blob-storing-database/src/Volo.Abp.BlobStoring.Database.Domain/Volo/Abp/BlobStoring/Database/DatabaseBlobProvider.cs
public class DatabaseBlobProvider : BlobProviderBase, ITransientDependency
{
    protected IDatabaseBlobRepository DatabaseBlobRepository { get; }
    protected IDatabaseBlobContainerRepository DatabaseBlobContainerRepository { get; }
    protected IGuidGenerator GuidGenerator { get; }
    protected ICurrentTenant CurrentTenant { get; }

    public DatabaseBlobProvider(
        IDatabaseBlobRepository databaseBlobRepository,
        IDatabaseBlobContainerRepository databaseBlobContainerRepository,
        IGuidGenerator guidGenerator,
        ICurrentTenant currentTenant)
    {
        DatabaseBlobRepository = databaseBlobRepository;
        DatabaseBlobContainerRepository = databaseBlobContainerRepository;
        GuidGenerator = guidGenerator;
        CurrentTenant = currentTenant;
    }
    // ...
}
```

### Save

`SaveAsync` does three jobs: lazily creates the container row if it does not exist, reads the bytes from the stream, then either inserts or updates the blob row depending on whether one already exists:

```csharp
// modules/blob-storing-database/src/Volo.Abp.BlobStoring.Database.Domain/Volo/Abp/BlobStoring/Database/DatabaseBlobProvider.cs
public async override Task SaveAsync(BlobProviderSaveArgs args)
{
    var container = await GetOrCreateContainerAsync(args.ContainerName, args.CancellationToken);

    var blob = await DatabaseBlobRepository.FindAsync(
        container.Id, args.BlobName, args.CancellationToken);

    var content = await args.BlobStream.GetAllBytesAsync(args.CancellationToken);

    if (blob != null)
    {
        if (!args.OverrideExisting)
        {
            throw new BlobAlreadyExistsException(
                $"Saving BLOB '{args.BlobName}' does already exists in the container '{args.ContainerName}'! " +
                $"Set {nameof(args.OverrideExisting)} if it should be overwritten.");
        }

        blob.SetContent(content);
        await DatabaseBlobRepository.UpdateAsync(blob, autoSave: true);
    }
    else
    {
        blob = new DatabaseBlob(GuidGenerator.Create(), container.Id, args.BlobName, content, CurrentTenant.Id);
        await DatabaseBlobRepository.InsertAsync(blob, autoSave: true);
    }
}
```

Two design choices stand out:

- **`GetAllBytesAsync`** — the entire stream is buffered into memory before the database call. For very large blobs (multi-GB), the database provider is not ideal; consider the file system or cloud provider instead.
- **`autoSave: true`** — each save commits immediately rather than waiting for the outer Unit of Work. This is intentional: a blob upload is usually independent of any business transaction, and `autoSave` keeps the open connection time minimal.

The `BlobAlreadyExistsException` thrown when `OverrideExisting == false` mirrors the contract every blob provider must honour — see [Blob Storing](/infrastructure/blob-storing).

### Delete / Exists / Get

The other three operations follow the same pattern: locate the container first, return `false` / `null` if the container does not exist, otherwise delegate to the blob repository:

```csharp
// modules/blob-storing-database/src/Volo.Abp.BlobStoring.Database.Domain/Volo/Abp/BlobStoring/Database/DatabaseBlobProvider.cs
public async override Task<bool> DeleteAsync(BlobProviderDeleteArgs args)
{
    var container = await DatabaseBlobContainerRepository.FindAsync(args.ContainerName, args.CancellationToken);
    if (container == null) { return false; }

    return await DatabaseBlobRepository.DeleteAsync(
        container.Id, args.BlobName, autoSave: true, cancellationToken: args.CancellationToken);
}

public async override Task<bool> ExistsAsync(BlobProviderExistsArgs args)
{
    var container = await DatabaseBlobContainerRepository.FindAsync(args.ContainerName, args.CancellationToken);
    if (container == null) { return false; }

    return await DatabaseBlobRepository.ExistsAsync(container.Id, args.BlobName, args.CancellationToken);
}

public async override Task<Stream> GetOrNullAsync(BlobProviderGetArgs args)
{
    var container = await DatabaseBlobContainerRepository.FindAsync(args.ContainerName, args.CancellationToken);
    if (container == null) { return null; }

    var blob = await DatabaseBlobRepository.FindAsync(container.Id, args.BlobName, args.CancellationToken);
    if (blob == null) { return null; }

    return new MemoryStream(blob.Content);
}
```

`GetOrNullAsync` returns a `MemoryStream` wrapping the byte array — callers should `await using` the result. The byte array is shared by reference, so do *not* mutate it after the call.

### GetOrCreateContainerAsync

The container is materialised on first save against a name. There is no separate "register container" call — the first `SaveAsync` for a container name inserts the row:

```csharp
// modules/blob-storing-database/src/Volo.Abp.BlobStoring.Database.Domain/Volo/Abp/BlobStoring/Database/DatabaseBlobProvider.cs
protected virtual async Task<DatabaseBlobContainer> GetOrCreateContainerAsync(
    string name, CancellationToken cancellationToken = default)
{
    var container = await DatabaseBlobContainerRepository.FindAsync(name, cancellationToken);
    if (container != null) { return container; }

    container = new DatabaseBlobContainer(GuidGenerator.Create(), name, CurrentTenant.Id);
    await DatabaseBlobContainerRepository.InsertAsync(container, cancellationToken: cancellationToken);

    return container;
}
```

<Note>
The container is created with `CurrentTenant.Id`. Because the container is multi-tenant, two tenants saving to the same container name end up with *separate* `DatabaseBlobContainer` rows that share a name but have different `TenantId` values. The `(TenantId, Name)` index on the table enforces uniqueness within a tenant scope.
</Note>

## Wiring a container to the database provider

The module ships a one-liner extension that points a `BlobContainerConfiguration` at `DatabaseBlobProvider`:

```csharp
// modules/blob-storing-database/src/Volo.Abp.BlobStoring.Database.Domain/Volo/Abp/BlobStoring/Database/DatabaseBlobContainerConfigurationExtensions.cs
public static class DatabaseBlobContainerConfigurationExtensions
{
    public static BlobContainerConfiguration UseDatabase(
        this BlobContainerConfiguration containerConfiguration)
    {
        containerConfiguration.ProviderType = typeof(DatabaseBlobProvider);
        return containerConfiguration;
    }
}
```

`BlobStoringDatabaseDomainModule.ConfigureServices` opts every *default-configured* container into database storage. The condition `container.ProviderType == null` means hosts that already chose a specific provider for a container are not overridden:

```csharp
// modules/blob-storing-database/src/Volo.Abp.BlobStoring.Database.Domain/Volo/Abp/BlobStoring/Database/BlobStoringDatabaseDomainModule.cs
[DependsOn(
    typeof(AbpDddDomainModule),
    typeof(AbpBlobStoringModule),
    typeof(BlobStoringDatabaseDomainSharedModule)
    )]
public class BlobStoringDatabaseDomainModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        Configure<AbpBlobStoringOptions>(options =>
        {
            options.Containers.ConfigureDefault(container =>
            {
                if (container.ProviderType == null)
                {
                    container.UseDatabase();
                }
            });
        });
    }
}
```

A host can override per container:

```csharp
Configure<AbpBlobStoringOptions>(options =>
{
    options.Containers.Configure<ProfilePictureContainer>(container =>
    {
        container.UseDatabase(); // explicit, even if defaults change
    });

    options.Containers.Configure<LargeFileContainer>(container =>
    {
        container.UseFileSystem(fs => fs.BasePath = "/srv/blobs");
    });
});
```

This pattern — global default + per-container override — is exactly the registration model used by `IBlobContainerFactory<T>` and described in [Blob Storing](/infrastructure/blob-storing).

## Entity Framework Core mapping

`BlobStoringDbContextModelCreatingExtensions.ConfigureBlobStoring` declares two tables, `AbpBlobContainers` and `AbpBlobs`, with a foreign-key relationship and an index that powers the `(TenantId, ContainerId, Name)` lookups:

```csharp
// modules/blob-storing-database/src/Volo.Abp.BlobStoring.Database.EntityFrameworkCore/Volo/Abp/BlobStoring/Database/EntityFrameworkCore/BlobStoringDbContextModelCreatingExtensions.cs
public static void ConfigureBlobStoring(this ModelBuilder builder)
{
    Check.NotNull(builder, nameof(builder));

    builder.Entity<DatabaseBlobContainer>(b =>
    {
        b.ToTable(AbpBlobStoringDatabaseDbProperties.DbTablePrefix + "BlobContainers",
            AbpBlobStoringDatabaseDbProperties.DbSchema);

        b.ConfigureByConvention();

        b.Property(p => p.Name).IsRequired().HasMaxLength(DatabaseContainerConsts.MaxNameLength);

        b.HasMany<DatabaseBlob>().WithOne().HasForeignKey(p => p.ContainerId);

        b.HasIndex(x => new { x.TenantId, x.Name });

        b.ApplyObjectExtensionMappings();
    });

    builder.Entity<DatabaseBlob>(b =>
    {
        b.ToTable(AbpBlobStoringDatabaseDbProperties.DbTablePrefix + "Blobs",
            AbpBlobStoringDatabaseDbProperties.DbSchema);

        b.ConfigureByConvention();

        b.Property(p => p.ContainerId).IsRequired();
        b.Property(p => p.Name).IsRequired().HasMaxLength(DatabaseBlobConsts.MaxNameLength);
        b.Property(p => p.Content).HasMaxLength(DatabaseBlobConsts.MaxContentLength);

        b.HasOne<DatabaseBlobContainer>().WithMany().HasForeignKey(p => p.ContainerId);

        b.HasIndex(x => new { x.TenantId, x.ContainerId, x.Name });

        b.ApplyObjectExtensionMappings();
    });

    builder.TryConfigureObjectExtensions<BlobStoringDbContext>();
}
```

The `HasMany<DatabaseBlob>().WithOne()` configuration uses unowned navigation — `DatabaseBlob` is itself an aggregate root and is not loaded as a child of `DatabaseBlobContainer`. The relationship exists purely to model the foreign key; queries against blobs go through `IDatabaseBlobRepository`, not through navigation properties.

`EfCoreDatabaseBlobRepository` implements the simple by-name lookup queries:

```csharp
// modules/blob-storing-database/src/Volo.Abp.BlobStoring.Database.EntityFrameworkCore/Volo/Abp/BlobStoring/Database/EntityFrameworkCore/EfCoreDatabaseBlobRepository.cs
public virtual async Task<DatabaseBlob> FindAsync(
    Guid containerId, string name, CancellationToken cancellationToken = default)
{
    return (await GetDbSetAsync())
        .FirstOrDefault(x => x.ContainerId == containerId && x.Name == name);
}

public virtual async Task<bool> ExistsAsync(
    Guid containerId, string name, CancellationToken cancellationToken = default)
{
    return await (await GetDbSetAsync())
        .AnyAsync(x => x.ContainerId == containerId && x.Name == name,
            GetCancellationToken(cancellationToken));
}

public virtual async Task<bool> DeleteAsync(
    Guid containerId, string name, bool autoSave = false,
    CancellationToken cancellationToken = default)
{
    var blob = await FindAsync(containerId, name, cancellationToken);
    if (blob == null) { return false; }

    await base.DeleteAsync(blob, autoSave, cancellationToken: GetCancellationToken(cancellationToken));
    return true;
}
```

`EfCoreDatabaseBlobContainerRepository.FindAsync(string name, ...)` is the matching by-name lookup for containers, returning the row whose `Name` matches in the current tenant scope (the `[TenantId]` data filter automatically narrows the result).

See [Entity Framework Core](/data/entity-framework-core) for `EfCoreRepository<,,>`, `ConfigureByConvention()` and the object-extension mappings.

## MongoDB mapping

The Mongo registration mirrors EF — two collections, `AbpBlobContainers` and `AbpBlobs`:

```csharp
// modules/blob-storing-database/src/Volo.Abp.BlobStoring.Database.MongoDB/Volo/Abp/BlobStoring/Database/MongoDB/BlobStoringMongoDbContextExtensions.cs
public static void ConfigureBlobStoring(this IMongoModelBuilder builder)
{
    Check.NotNull(builder, nameof(builder));

    builder.Entity<DatabaseBlobContainer>(b =>
    {
        b.CollectionName = AbpBlobStoringDatabaseDbProperties.DbTablePrefix + "BlobContainers";
    });

    builder.Entity<DatabaseBlob>(b =>
    {
        b.CollectionName = AbpBlobStoringDatabaseDbProperties.DbTablePrefix + "Blobs";
    });
}
```

`MongoDbDatabaseBlobRepository.FindAsync` uses the asynchronous LINQ provider over the underlying `IMongoCollection<DatabaseBlob>`:

```csharp
// modules/blob-storing-database/src/Volo.Abp.BlobStoring.Database.MongoDB/Volo/Abp/BlobStoring/Database/MongoDB/MongoDbDatabaseBlobRepository.cs
public virtual async Task<DatabaseBlob> FindAsync(Guid containerId, string name,
    CancellationToken cancellationToken = default)
{
    cancellationToken = GetCancellationToken(cancellationToken);

    return await (await GetQueryableAsync(cancellationToken))
        .FirstOrDefaultAsync(x => x.ContainerId == containerId && x.Name == name, cancellationToken);
}
```

The container lookup is similarly simple:

```csharp
// modules/blob-storing-database/src/Volo.Abp.BlobStoring.Database.MongoDB/Volo/Abp/BlobStoring/Database/MongoDB/MongoDbDatabaseBlobContainerRepository.cs
public virtual async Task<DatabaseBlobContainer> FindAsync(string name,
    CancellationToken cancellationToken = default)
{
    return await base.FindAsync(x => x.Name == name,
        cancellationToken: GetCancellationToken(cancellationToken));
}
```

<Warning>
MongoDB documents are capped at 16 MB. If you need to store larger blobs in Mongo, use the GridFS-backed provider rather than this module — or shard your storage by routing large blobs to a different `BlobContainerConfiguration`. See [MongoDB Integration](/data/mongodb-integration).
</Warning>

## Multi-tenancy and data isolation

Both aggregates implement `IMultiTenant`. The combined effect is:

- A tenant's blobs are filtered out automatically by the `[TenantId]` data filter — `DatabaseBlobRepository.FindAsync` in tenant context never returns a host row, and vice versa.
- The `(TenantId, ContainerId, Name)` index guarantees that two tenants can store a blob with the same `(container, name)` pair without collision.
- The container is created on demand with `CurrentTenant.Id`, so a "first save by a tenant" creates a tenant-scoped container row.

This means a tenant database can either share the host's `AbpBlobs` table (multi-tenant table) or run an isolated one (tenant-only DB) — the same code path works in both because the data filter is the only thing that changes. See [Multi-Tenancy Overview](/multi-tenancy/overview) for the data-filter mechanism.

## End-to-end flow

```csharp
// host code
public class ProfilePictureService : ApplicationService
{
    private readonly IBlobContainer<ProfilePictureContainer> _profilePictures;

    public ProfilePictureService(IBlobContainer<ProfilePictureContainer> profilePictures)
    {
        _profilePictures = profilePictures;
    }

    public async Task UploadAsync(Guid userId, Stream content)
    {
        await _profilePictures.SaveAsync(userId.ToString(), content, overrideExisting: true);
    }
}
```

Behind that call:

1. `IBlobContainer<ProfilePictureContainer>` resolves to a `BlobContainer` whose configuration points `ProviderType = typeof(DatabaseBlobProvider)`.
2. `BlobContainer.SaveAsync` calls `DatabaseBlobProvider.SaveAsync` with `args.ContainerName = "ProfilePictureContainer"` and `args.BlobName = userId.ToString()`.
3. `DatabaseBlobProvider.GetOrCreateContainerAsync` resolves or creates the `DatabaseBlobContainer` row.
4. The stream bytes are buffered and persisted as either an insert or an update on `DatabaseBlob`.

## Summary

The Blob Storing Database module is one of the simplest blob providers in the ABP ecosystem: two aggregate roots (`DatabaseBlobContainer`, `DatabaseBlob`), a four-method `DatabaseBlobProvider`, and parallel EF Core / MongoDB persistence. The `UseDatabase()` extension and the default-container hook in `BlobStoringDatabaseDomainModule` make it the no-config provider for any container that hasn't been wired to a more specialised backend. Because the bytes live inline in the database, a single backup captures the entire blob store — a real operational benefit for small-to-medium content workloads.
